# AI-first vibe coding 脚手架产品能力地图 + MVP 范围定义

## 1. 文档信息

- 文档名称：AI-first vibe coding 脚手架产品能力地图 + MVP 范围定义
- 文档目标：统一产品定位、能力边界、MVP 范围与实施优先级，为下一步系统架构、数据模型和工程实现提供基线
- 适用阶段：产品定义、方案设计、0 到 1 实现规划
- 当前版本：v0.1
- 状态：可进入架构设计与工程拆解

---

## 2. 背景与问题定义

当前主流 AI 编程工具已经具备代码生成、代码修改、局部分析和基础 agent 能力，但对研发团队而言，仍然存在几个关键问题：

1. 存量研发人员并不天然熟悉 AI 编程范式，容易停留在“问一句、改一点”的浅层使用方式。
2. 新项目和老项目的接入方式不统一，老项目尤其缺少稳定的 AI 接管路径。
3. 前端、后端、全栈、文档、安全、测试等活动彼此割裂，AI 能力没有形成完整交付闭环。
4. 多人和多 agent 并行改动时，缺少任务边界、冲突治理和统一检查机制。
5. 代码在持续变化，但项目知识、wiki、标准、经验沉淀很容易过期。
6. 团队常有大量已有经验、模板、规范和 skill，但无法在项目推进过程中被稳定调用与持续进化。
7. 团队实际使用的 AI coding tools 可能不同，若平台绑定单一工具，将显著降低 adoption。

本项目要解决的核心问题不是“如何让 AI 写更多代码”，而是：

**如何把研发过程固化成一种 AI-first 的 vibe coding 范式，让普通研发把项目接入后，就能被持续引导、被检查、被沉淀、被协同。**

---

## 3. 产品定位

### 3.1 一句话定位

这是一个面向 `vibe coding` 范式的 **AI 引导式研发控制层 / 脚手架平台**，用于在新项目和老项目中统一项目理解、阶段判断、下一步引导、执行检查、知识同步与团队经验沉淀。

### 3.2 它不是什么

- 不是只会生成代码的 prompt 工具
- 不是单一 AI coding tool 的绑定插件
- 不是要求项目必须重构后才能接入的平台
- 不是只服务新项目的 starter template

### 3.3 它是什么

- 是项目级 AI 控制层
- 是研发过程引导系统
- 是 agent + skill 的编排运行面 —— **Harness：记忆、skill、子智能体**
- 是团队管理与协作治理机制
- 是团队经验沉淀与同步机制
- 是跨 AI coding tools 的中立适配层

---

## 4. 产品目标

### 4.1 主目标

1. 让不熟悉 AI 编程范式的研发人员也能直接上手。
2. 让新项目和老项目都能被平台接住并持续推进。
3. 始终让用户知道”当前项目在哪个阶段、下一步最应该做什么”。
4. 让每个关键执行动作都有检查机制兜底。
5. 让项目实践持续沉淀为项目知识、团队标准和可复用 skill。
6. 让多人协作与全栈交付在同一套 AI 引导范式下运作。
7. 让平台具备对主流 AI coding tools 的理解、感知和适配能力。
8. **让团队有清晰的角色分工、任务归属、决策留痕和节奏推进机制。**
9. **让Harness特性（记忆、skill、子智能体）成为可复用的团队资产。**

### 4.2 成功标志

若产品达到以下状态，可视为第一阶段方向正确：

1. 新项目可在 30 分钟内完成初始化并拿到第一轮明确引导。
2. 老项目可在不改变原始代码结构的前提下完成接入和首轮诊断。
3. 任一时刻用户都可以通过统一入口看到当前阶段、建议动作、风险和推荐执行角色。
4. 每次关键代码变更后，系统能发现知识可能过期并生成同步建议。
5. 团队可在项目中复用本地 skill 和外部技能市场 skill。
6. 同一项目可在至少 1 到 2 个主流 AI coding tools 环境中工作。
7. **团队可以清晰看到谁负责什么、决策是如何做出的、下一步节奏是什么。**
8. **子智能体可以分治简单任务，记忆可以在会话和项目级别沉淀和复用。**

## 5. 核心设计原则

1. 项目接入优先于代码生成。
2. 引导优先于自由发挥。
3. 检查必须内建，不能可有可无。
4. 经验沉淀必须可持续更新，而不是静态文档堆积。
5. 老项目要渐进接管，不强制重构。
6. 全栈交付按业务闭环组织，不按技术栈割裂。
7. 多人协作靠机制治理，不靠口头约定。
8. 平台能力必须对 AI coding tools 保持中立。
9. skill 是能力模块，不是散落 prompt。
10. 自动化必须有手动兜底入口。

## 6. 目标用户与典型场景

### 6.1 目标用户

1. 对 AI 编程范式不熟悉，但希望开始使用 AI 提升效率的存量研发人员
2. 希望把前端、后端经验沉淀进统一工程范式的技术负责人
3. 正在推动团队往全栈交付演进的研发团队
4. 需要统一新项目和老项目接入体验的组织

### 6.2 典型场景

1. 新项目从 idea 开始，平台引导完成需求设计、方案设计、脚手架生成和开发推进。
2. 老项目直接接入，平台自动完成代码、安全、逻辑、文档、wiki、优化建议扫描。
3. 用户可直接提供现有需求文档，平台给出建议，用户可采纳也可不采纳。
4. 一位研发同时负责已有前后端项目的某个业务功能闭环，平台统一引导其前后端交付。
5. 多位研发同时迭代老项目，平台通过任务边界和检查机制降低冲突。
6. 团队在使用中逐步把经验沉淀成标准、模板和 skill。
7. 团队在不同 AI coding tools 中保持一致的项目推进方式。

## 7. 产品边界

### 7.1 本项目负责什么

1. 项目接入与项目识别
2. 阶段判断与下一步引导
3. agent / skill 调度与执行治理
4. **团队管理：角色分工、任务归属、决策留痕、节奏推进**
5. **Harness 特性：记忆管理、子智能体调度、技能编排**
6. 检查、review、扫描与发布前 gate
7. 项目知识、wiki、标准和经验沉淀
8. 多人协作中的变更协调和冲突治理
9. AI coding tools 的能力适配

### 7.2 本项目暂不直接负责什么

1. 取代业务框架本身
2. 替代源代码托管平台
3. 提供重量级组织管理后台
4. 一开始就做完全部第三方生态深度集成

## 8. 产品能力地图

本项目能力地图按**十大能力域**组织。

### 8.1 项目接入能力

目标：让新项目和老项目都能低阻力进入平台。

能力项：

1. 新项目初始化
2. 老项目接管
3. 项目类型识别
4. 全栈老项目统一注入
5. 项目根目录 AI 控制层注入

说明：

- 新项目采用托管式接入
- 老项目采用伴生式接入
- 老项目全栈场景下只在项目根目录注入一次 `.ai-first/`

### 8.2 项目理解与阶段判断能力

目标：让平台始终知道项目当前在哪里。

能力项：

1. 技术栈识别
2. 项目结构扫描
3. 文档与测试完备度判断
4. 当前阶段识别
5. 风险、阻塞和缺失产物识别
6. 新老项目模式判断
7. 团队协作模式判断

输出应形成统一的 `Project Snapshot`。

### 8.3 下一步引导能力

目标：让用户始终知道下一步最应该做什么。

能力项：

1. 当前阶段说明
2. 建议下一动作
3. 可选替代动作
4. 风险提示
5. 推荐主导 agent
6. 可直接执行的操作入口
7. 支持生成、复用、跳过三种推进方式

输出应形成统一的 `Guidance Card`。

### 8.4 执行编排能力

目标：让不同 agent 和 skill 能在统一协议下推进任务。

能力项：

1. 主导 agent 选择
2. **子智能体调度与分治执行** —— 支持任务拆分、并行执行、结果聚合
3. skill 自动匹配
4. 执行步骤管理
5. 任务状态推进
6. 前后端闭环任务拆解
7. **执行上下文记忆管理** —— 会话级、项目级、混合记忆模式

### 8.5 检查与治理能力

目标：让所有关键动作都有检查角色兜底。

能力项：

1. 代码 review
2. 逻辑 bug 扫描
3. 安全扫描
4. 架构风险检查
5. 文档完备度检查
6. 知识同步检查
7. 发布前 gate

### 8.6 知识沉淀与同步能力

目标：让代码变化后，项目知识和团队经验不会失真。

能力项：

1. 项目知识库维护
2. wiki 构建与更新
3. 决策日志维护
4. 标准候选项提炼
5. skill 候选项提炼
6. 关键变更自动触发同步
7. 手动同步兜底入口

### 8.7 多人协作与防冲突能力

目标：让多人迭代场景可治理、可追踪、可收敛。

能力项：

1. 任务边界定义 — 按 `CodeDomain` 分派任务，天然隔离
2. change scope 声明 — 透明化各自影响范围，提前发现冲突
3. 并行变更冲突提示 — `ChangeScope` 重叠时发出预警
4. 影响方提示 — 基于路径匹配提示可能受影响的任务，由人工判断冲突
5. Git 原生冲突解决 — 分支隔离、PR review、merge 处理代码冲突
6. owner / reviewer 双角色机制 — 责任清晰，检查兜底

### 8.8 团队管理能力

目标：让团队有清晰的角色分工、任务归属、决策留痕和节奏推进机制。

能力项：

1. **角色分工与职责定义** — 明确 Intake/Planner/Architect/Builder/Reviewer/Release/TeamLead 等角色的职责边界
2. **任务归属与状态追踪** — 任务 owner、reviewer 双重责任，任务状态全程可追溯
3. **决策留痕与可回溯** — 技术决策、架构决策、跳过决策都需记录原因和风险
4. **节奏推进与里程碑管理** — 支持按阶段、按迭代、按里程碑推进，而非散乱的改动
5. **团队知识共享** — 决策、经验、pattern 在团队内可复用
6. **协作冲突解决机制** — 冲突检测、协商建议、解决记录

说明：本能力域与"多人协作与防冲突"的区别在于——协作治理侧重**单次执行的协调**，团队管理侧重**持续的组织能力建设**。

### 8.9 skill 生态能力

目标：让平台既能复用团队经验，也能复用技能市场能力。

能力项：

1. 本地 skill 管理
2. 团队 skill 沉淀
3. 技能市场 skill 接入
4. skill 协议统一
5. skill 风险治理
6. skill 使用效果反馈

### 8.10 AI coding tools 适配能力

目标：让平台不绑定单一工具，而成为统一控制层。

能力项：

1. 工具环境识别
2. 工具能力画像
3. 上下文注入适配
4. slash 命令映射
5. skill 调用适配
6. 输出归一化
7. 工具差异下的执行策略选择

目标兼容方向包括但不限于：

- Codex
- Claude Code
- Trae
- Qoder

## 9. 典型工作流

### 9.1 新项目工作流

1. 用户输入项目 idea
2. 平台初始化项目
3. 平台识别项目类型和建议起步阶段
4. 用户选择：
   - 让平台生成需求
   - 提供已有需求文档
   - 跳过需求阶段
5. 平台输出当前阶段与下一步建议
6. 主导 agent 推进任务
7. reviewer 检查执行结果
8. 知识与标准同步更新

### 9.2 老项目工作流

1. 用户将老项目接入平台
2. 平台在项目根目录注入 `.ai-first/`
3. 平台扫描前端、后端、文档、配置、测试
4. 平台识别当前阶段、问题和缺口
5. 平台输出诊断结果与下一步建议
6. 用户选择继续执行的目标
7. 主导 agent 推进
8. reviewer 检查
9. 同步 wiki、知识和经验沉淀

### 9.3 老项目全栈闭环工作流

1. 平台把前后端作为同一项目的两个代码域识别
2. 任务按业务功能闭环定义，而不是单纯按前端 / 后端分拆
3. 平台为任务建立统一 change scope
4. 用户按完整交付链路推进前后端改动
5. reviewer 检查接口契约、逻辑、测试、知识同步

## 10. MVP 定义

### 10.1 MVP 目标

第一版不是做全量平台，而是打通最小闭环：

**项目接入 -> 项目识别 -> 下一步引导 -> 执行推进（记忆+子智能体+skill） -> 检查兜底 -> 知识同步 -> 团队管理**

### 10.2 MVP 必须完成的能力

#### A. 接入与识别

1. 支持新项目初始化
2. 支持老项目接入
3. 支持项目根目录注入 `.ai-first/`
4. 支持技术栈和项目结构基础识别
5. 支持当前阶段基础判断

#### B. 引导与推进

1. 支持统一输出当前阶段与下一步建议
2. 支持生成、复用、跳过三种阶段推进模式
3. 支持基础任务创建和状态推进
4. 支持新项目和老项目两类主流程

#### C. 扫描与检查

1. 支持基础安全扫描
2. 支持基础逻辑 bug / 代码问题扫描
3. **支持优化建议输出** — 老项目接入后必须给出面向人的优化建议（结构、性能、可维护性等）
4. 支持基础文档 / wiki 构建能力
5. 支持 reviewer 角色检查
6. 支持知识同步检查

#### D. 知识同步

1. 支持关键变更触发知识同步建议
2. 支持 wiki 更新建议
3. 支持项目知识写回
4. 支持 `/sync` 手动兜底

#### E. 协作治理

1. 支持任务 owner / reviewer
2. 支持基础 change scope
3. 支持基础冲突提示

#### F. skill 与工具适配

1. 支持本地 skill 基础接入
2. 支持技能市场 skill 基础接入
3. 支持统一 skill 元数据协议
4. 支持 1 到 2 个 AI coding tools 适配器

#### G. Harness 核心特性

1. **子智能体调度** — 支持任务拆分后分派给不同角色 agent，支持简单并行执行和结果聚合
2. **记忆管理** — 支持会话级记忆（单次对话）、项目级记忆（`.ai-first/knowledge/`），暂不支持跨项目记忆
3. **技能编排** — skill 声明适用的阶段/任务类型，系统做推荐匹配，MVP 不做严格的协议验证

说明：MVP 阶段的 Harness 是"可用单机版"，不是"分布式编排平台"。重点是证明：
- 记忆可以沉淀和复用
- skill 可以稳定调度
- 子智能体可以分治简单任务

### 10.3 MVP 暂不覆盖或降级覆盖的能力

1. 复杂 Web 管理台
2. 大规模多 agent 并发编排
3. 组织级审批与细粒度权限体系
4. 完整企业级报表和运营后台
5. 所有 AI coding tools 的深度适配
6. 高级自动修复和全自动发布编排

## 12. 用户可见入口设计

第一版建议优先提供统一斜杠命令入口。

### 12.1 基础命令

- `/init`：初始化新项目 AI 控制层
- `/adopt`：接管老项目
- `/guide`：查看当前阶段、建议动作和风险
- `/scan`：重新扫描项目状态
- `/sync`：手动触发知识同步
- `/wiki`：重建或更新 wiki
- `/review`：触发 reviewer 检查
- `/standards`：查看或更新标准
- `/skills`：查看启用 skill 和推荐 skill

### 12.2 命令设计原则

1. 用户日常主入口是 `/guide`
2. `/sync` 必须作为知识更新保险丝存在
3. 所有命令输出都应回归统一 `Guidance Card` 或 `Review Report`

## 13. 关键机制说明

### 13.1 多人迭代冲突防护机制

目标：确保多人同时迭代同一项目时，代码冲突可预测、可治理、可收敛。

**机制 1：Change Scope 声明**
- 每个任务执行前必须声明 `ChangeScope`：包含前端路径、后端路径、共享路径、文档路径
- `ChangeScope` 存储在 `.ai-first/change-scopes/` 下，可被其他成员查看

**机制 2：任务边界隔离**
- **任务分配时隔离** — 同一模块/域的任务优先分配给同一 owner，减少并行冲突
- **按域分派** — 前端/后端/共享模块的任务按 `CodeDomain` 分派，天然隔离
- **风险提示** — `ChangeScope` 重叠时发出警告，但**不强制阻止**，由团队自行协调

**机制 3：冲突检测**
- 检测维度：
  1. **路径重叠检测** — 两个任务的 `ChangeScope` 是否有路径交集
  2. **影响方提示** — 基于 `ChangeScope` 路径匹配，列出可能受影响的其他任务，由人工判断是否需要协调
- 检测时机：任务创建时、代码提交时、review 时

**机制 3：Git 原生冲突解决**
- **代码冲突由 Git 处理** — 分支隔离、PR review、merge 冲突解决
- **ChangeScope 作为提前预警** — 在写代码前就知道可能冲突，而非 PR 时才发现
- **Reviewer 兜底检查** — 有未解决冲突时 `ReviewReport` 状态为 `failed`，阻塞合并

**协作原则**：
1. 优先通过任务分配隔离冲突
2. ChangeScope 作为透明度工具，而非强制锁
3. 代码冲突信任 Git + PR 流程
4. 代码冲突由 Git 处理，任务协调由人工完成

**机制 5：Reviewer 兜底**
- reviewer 检查时必须验证：是否与正在进行的其他任务有冲突
- 如有未处理的冲突，`ReviewReport` 状态为 `failed`

### 13.2 知识同步触发与更新机制

目标：确保代码变更后，相关项目知识能被及时识别和更新。

**机制 1：同步触发条件**
以下情况自动生成 `SyncEvent`：

| 触发条件 | 同步内容 | 优先级 |
|---------|---------|--------|
| 修改 API 接口/契约 | api-contract 知识项 + fullstack 标准 | 高 |
| 修改数据模型/Schema | 数据模型知识项 + 相关 wiki | 高 |
| 修改架构/模块结构 | 架构知识项 + DomainMap | 高 |
| 修改安全相关代码 | 安全标准 + 知识项 | 高 |
| 完成 feature/bugfix | 相关文档/示例/决策记录 | 中 |
| 修改配置/部署 | 部署知识项 + 运维标准 | 中 |
| 阶段退出（spec → architecture） | 下一阶段知识候选项 | 中 |

- 触发后生成 `SyncEvent`，状态为 `suggested`，存储在 `.ai-first/sync/`

**机制 2：知识影响分析**
当变更发生时，系统分析：
1. **路径匹配** → 变更的文件路径是否在 `KnowledgeItem.relatedPaths` 中
2. **类型匹配** → 变更类型（API/Schema/架构/安全）是否匹配知识项类型
3. **简单触发** → 匹配成功则生成同步建议，由 reviewer 确认是否真的需要更新

**机制 3：同步确认流程**
```
1. 代码变更 → 自动生成 SyncEvent（status: suggested）
2. Reviewer 检查 → 确认需要同步
3. 用户/AI 生成更新内容 → 草稿写入 `.ai-first/knowledge/`
4. 人工确认 → status: confirmed，正式覆盖原内容
5. 人工驳回 → status: dismissed，记录原因
```

**机制 4：手动同步兜底**
- 用户随时可执行 `/sync` 命令
- `/sync` 扫描最近变更，主动生成同步建议
- 用于：自动同步遗漏、批量更新、定期同步维护

**机制 5：知识过期检测**
- 定期扫描：知识项引用的路径/代码是否已变更
- 过期标记：`KnowledgeItem.stability` 标记为 `deprecated`
- 过期提示：`/guide` 输出中包含过期知识警告

## 14. 老项目全栈接入约束

这是本项目的关键兼容约束之一。

1. 老项目接入时，只在项目根目录注入一个 `.ai-first/`
2. 不为前端和后端分别创建两个独立 AI 项目
3. 项目内前后端作为同一 `Project` 下的两个代码域管理
4. 任务按业务功能闭环组织
5. reviewer 要检查前后端契约一致性
6. 标准沉淀中要包含 `fullstack` 视角

## 15. 风险与非功能要求

### 15.1 关键风险

1. 过早做成重平台，导致 0 到 1 过慢
2. 过度绑定单一 AI coding tool，降低 adoption
3. 知识同步机制不足，导致沉淀快速失效
4. 老项目接入成本过高，导致团队不愿迁移
5. 多人协作机制缺失，导致 agent 与人并行时冲突增加

### 15.2 非功能要求

1. 接入老项目时尽量零侵入
2. 输出必须结构化、可追踪、可落盘
3. 所有关键协议必须工具中立
4. 重要自动化流程必须支持手动重触发
5. 经验沉淀必须区分临时信息、项目知识和团队标准

## 17. 当前结论

本项目第一阶段的正确路线不是”做一个更强的 AI 写代码工具”，而是：

**做一个面向新老项目、支持全栈闭环、兼容主流 AI coding tools、可持续沉淀团队经验、具备团队管理能力和 Harness 核心特性的 AI 引导式研发脚手架平台。**

MVP 只需要先打通最小闭环，但这个闭环必须完整：

**接入、识别、引导、执行（记忆+子智能体+skill）、检查、同步、团队管理，一个都不能缺。**
