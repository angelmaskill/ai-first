## 0. 总体判断

**方案本身合理、方向收敛正确，但能否"满足业务目标"取决于三个未在清单里显式处理的前提。**

清单是一份高质量的工程规划（优先级、批次、验收标准都到位），但它与本项目 README、CLAUDE.md 之间存在真实的产品定位张力，且最关键的 P0 项（Codex 执行闭环）风险被低估。

**一句话结论：值得按这份清单执行，但不要直接从 D2 开干。** 先化解"Claude-native vs 统一 CLI"的体验张力、给 Codex 输出做契约 fixture、拿真实项目手动跑一遍主链路——这三件事做完，清单的"剩余工作"才真正落地为可交付产品。

## 3. 能否满足业务目标：三个前提

清单第 20 节的 15 条"完成定义"与收敛后的业务目标自洽。但"满足业务目标"还差三个清单里没写明的前提。

### 前提 1：必须先化解"Claude-native vs 统一 CLI"的体验张力（最高风险）

- README 和 CLAUDE.md 的卖点：**Claude Code native、零命令、用户永远不需要知道 slash command、自然语言驱动**。
- 清单要做的事：`ai-first init/adopt/guide/task exec --runtime codex` 这种**确定性 CLI**，且 D2 明确要求"工具无关、不依赖 Claude"。

这两个方向不矛盾，但**体验定位不同**：一个重 LLM 编排，一个重确定性 + 跨工具。清单的 G 节（"Claude 指令调用共享核心"）承认了这件事，但力度不够——它只说"复用核心"，没回答：

> 做完之后，"零命令自然语言"这个 README 一级卖点还成立吗？如果 CLI 变成主入口，CLAUDE.md 那套 auto-orchestration 协议要怎么定位？

**建议：** 在第一批之前，先写一段"双形态体验定位"（CLI 是确定性骨架 / Claude 自然语言是增强层），并据此更新 README 和 CLAUDE.md 的措辞。否则做完会自相矛盾。

### 前提 2：F 节（Codex 执行闭环）是关键路径，但风险被低估

整个清单的价值落点就是 F——"Codex 能真实推进研发任务"。但 Codex 的**输出稳定性、长任务可靠性、结构化协议可解析性**是硬不确定项，而清单 F1–F4 把它当成普通功能列了出来。缺少：

- 如何**契约测试** Codex 是否真按 F3 协议输出（无 fixture、无 schema 校验计划）；
- "partial completion / parse failed"在 Codex 真实场景下的发生率与降级策略（F4 列了类型，但没说怎么验）。

**建议：** 把 F 提为"关键路径风险项"，在第一批里加一个 **F0：Codex 输出协议的契约测试 fixture**（哪怕手动跑 3 个真实小任务采集输出即可）。Codex 输出若不能稳定解析，整条主闭环就跑不通。

### 前提 3：缺一个真实试点项目锚定

L1 提了 fixture，但 fixture ≠ 真实用户。清单再合理，没有"一个真实 AI 项目从 init 到 evolve 跑一遍"的反馈，就只是空中楼阁。这正是"能否满足业务目标"最直接的检验。

**建议：** 在第一批之前插入**零号批次：拿一个真实的小型 AI 项目（前端 + 后端 + 算法），手动跑通 `adopt → guide → task → Codex exec → report → guide`**，用真实摩擦校准清单。本项目自己就是 self-hosting，正好 dogfooding。

## 8. 开干前建议处理的 5 个新问题

修订解决了旧问题，但引入/暴露了以下新风险，按重要性排序：

### N1. Z1 落地难度被低估：CLAUDE.md 是"重构心智模型"，不是措辞调整（高）

`.claude/CLAUDE.md` 现状是一套极详尽的 auto-orchestration 协议，核心假设是"用户无需知道命令、自然语言驱动、Claude 自动编排 + 自动推进阶段"。把它改成"CLI 是骨架、Claude 是增强层"，动摇的是该协议 auto-dispatch / auto-advance 的**行为基座**，而不是文档措辞。

Z1 验收只写"README、CLAUDE.md、使用指南都明确"，没回答关键行为裁决：

> 用户说"加暗黑模式"时，Claude 该走自己的 auto_chain，还是去调 `ai-first task exec`？auto-advance 还成立吗？

**建议：** Z1 拆出一条独立的"编排协议行为裁决"工作项，明确自然语言入口在双形态下的分发规则，再谈文档同步。

### N2. D2"无 LLM assessor"两层风险（高）

① **能力倒退**：纯规则/启发式判"项目处于 idea / discovery / spec"这类语义阶段，准确率通常低于 LLM，可能直接打击 §2.3 的"位置感"核心体验。清单未承认该 trade-off。
② **验收内部张力**：既要"纯规则函数"，又要输出"下一步建议"。纯规则对"阶段判定"够用，对"哪个阻塞最重要"的语义排序很弱。

**建议：** D2 拆两层——**阶段判定（纯规则、高置信）** + **阻塞优先级排序（规则给候选，LLM 可选增强）**，避免一刀切导致 guide 输出退化。可补一个 D0：拿若干真实项目样本测纯规则判阶段的准确率，低于阈值就回退为"规则 + LLM"。

### N3. F0 样本真实性 + F0/F2 循环依赖（高）

- F0 验收允许"真实**或半真实**样本"，但自己构造、自己解析无法证明对真实 Codex 输出的鲁棒性。**要求至少 2 个真实 Codex 输出。**
- **时序问题**：F0 排在 F2（prompt 模板）之前，但真实 Codex 输出依赖 prompt 模板。若先采样再定模板，样本会过时。**建议先定 F2 的"prompt v0 最小版"，F0 用 v0 采样**，或二者并行。

### N4. Z2"手动跑通"语义歧义（中）

`guide`(D3)、`task`(E1)、`task:exec`(F1) 尚未实现，Z2"跑通主链路"目前只能**手动模拟**。清单未点明，执行者可能误解为"等实现完再跑"，从而失去零号批次"提前暴露风险"的意义。

**建议：** Z2 描述明确为"以手动/脚本方式模拟尚未实现的环节，目标是验证协议层无硬阻塞、采集真实摩擦点"，而非端到端真实跑通。

### N5. 第一批缺"共享核心三层"约束 → G1 大返工风险（中）

第一批要实现的 `guide` / `task` / `task:exec`，正是 G1（第四批）要"复用共享核心"的命令。若第一批实现时不主动分层，第四批 G1 会变成大重构。

**建议：** 第一批加一条横切约束：**所有 guide/task/exec 从第一天就分 `core`（纯函数） / `cli`（入口） / `claude-command`（调 core）三层**。这本质是 §2.4 第4点（".ai-first 是共同上下文"）的工程落地，应显式化为工作项。
