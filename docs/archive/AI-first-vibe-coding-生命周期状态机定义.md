# AI-first vibe coding 生命周期状态机定义

## 1. 文档信息

- 文档名称：AI-first vibe coding 生命周期状态机定义
- 文档目标：定义项目生命周期状态机，为阶段判断、下一步引导、agent 路由、检查机制和知识同步提供统一基线
- 适用阶段：产品架构设计、核心流程设计、MVP 实现设计
- 当前版本：v0.1
- 状态：可进入数据模型和协议设计

---

## 2. 文档目的

本文件用于回答以下问题：

1. 平台如何判断项目当前处于哪个阶段
2. 每个阶段的进入条件和退出条件是什么
3. 每个阶段必须产出什么
4. 哪些阶段允许跳过、复用已有产物或由平台生成
5. 每个阶段应该由哪个主导 agent 推进
6. 每个阶段结束时应由什么检查角色兜底

该状态机不是为了把研发过程机械化，而是为了让平台在新项目、老项目、全栈交付和多人协作场景下，都能对“当前在哪、下一步做什么”给出一致判断。

---

## 3. 设计原则

1. 状态机服务于引导，不服务于流程管控本身。
2. 新项目和老项目使用同一套阶段模型，但进入点可以不同。
3. 每个阶段都支持 `Generate`、`Reuse`、`Skip` 三种处理方式。
4. 阶段之间允许前后回退，但每次回退都应记录原因。
5. 阶段判断应允许置信度存在，而不是强制二元判断。
6. 阶段完成不以“产物存在”为唯一标准，还要看检查是否通过。
7. 老项目通常从中间阶段切入，而不是从 `idea` 开始。

---

## 4. 生命周期总览

建议采用以下统一阶段模型：

1. `idea`
2. `discovery`
3. `spec`
4. `architecture`
5. `scaffold`
6. `build`
7. `qa`
8. `release`
9. `operate`
10. `evolve`

### 4.1 各阶段作用简述

- `idea`：定义项目意图、目标和边界
- `discovery`：澄清用户、场景、问题、约束和成功标准
- `spec`：形成需求说明、功能拆解和范围定义
- `architecture`：形成技术方案、模块边界和关键决策
- `scaffold`：准备工程骨架、基础规范和初始结构
- `build`：进入持续开发与迭代实现
- `qa`：进行质量检查、逻辑验证、安全扫描和回归确认
- `release`：准备上线、发布检查和交付确认
- `operate`：进入线上维护、反馈处理和运营支持
- `evolve`：基于运营、反馈和新目标进入下一轮迭代

### 4.2 状态流转主路径

标准主路径如下：

`idea -> discovery -> spec -> architecture -> scaffold -> build -> qa -> release -> operate -> evolve -> build`

说明：

1. `evolve` 不是终态，而是下一轮迭代的再规划入口
2. `build`、`qa`、`release`、`operate` 在老项目中通常会频繁循环
3. 老项目往往直接从 `build`、`qa`、`operate` 或 `evolve` 切入

## 5. 阶段定义

下面按阶段给出进入条件、核心目标、输入输出、可跳过规则、主导角色和检查机制。

### 5.1 `idea`

#### 阶段目标

明确为什么要做这个项目，它要解决什么问题。

#### 常见进入场景

1. 新项目从零开始
2. 用户只有一个模糊想法
3. 团队准备探索新的产品方向

#### 进入条件

满足以下任一条件即可：

1. 用户只提供一句想法或粗略方向
2. 项目尚无明确目标、范围和成功标准
3. 平台判断当前项目尚处于立项前思考阶段

#### 必备输入

1. 初始想法、业务方向或问题描述

#### 阶段输出

1. 项目目标草案
2. 问题定义
3. 初始边界和假设
4. 成功标准草案

#### 可跳过规则

以下情况可跳过到 `discovery` 或 `spec`：

1. 用户已提供较完整的目标与背景
2. 老项目接入，不需要重新定义项目意图

#### 主导 agent

- `Intake Agent`

#### 检查角色

- `Reviewer Agent`

#### 退出条件

满足以下条件可进入下一阶段：

1. 项目目标有初步明确表述
2. 问题与边界不再完全模糊
3. 平台能生成下一阶段问题清单

---

### 5.2 `discovery`

#### 阶段目标

理解目标用户、核心场景、约束条件、关键风险和成功标准。

#### 常见进入场景

1. 新项目已明确方向，但需求仍模糊
2. 老项目准备做大版本规划
3. 用户准备导入已有需求文档并让系统分析

#### 必备输入

1. 项目目标
2. 用户或业务背景
3. 已有需求资料或历史背景

#### 阶段输出

1. 目标用户与使用场景
2. 核心问题陈述
3. 约束和风险清单
4. 成功标准
5. 需求探索结论

#### 可处理方式

- `Generate`：平台生成调研与需求澄清骨架
- `Reuse`：用户上传已有需求材料，平台给建议
- `Skip`：允许跳过，但必须记录风险

#### 主导 agent

- `Planner Agent`

#### 检查角色

- `Reviewer Agent`

#### 退出条件

1. 主要用户、场景和问题范围已明确
2. 需求输入足以形成正式需求文档或功能拆解

---

### 5.3 `spec`

#### 阶段目标

形成明确的需求定义和功能范围。

#### 常见进入场景

1. 新项目准备进入正式设计
2. 用户提供 PRD、需求文档或 issue 列表
3. 老项目准备做某一业务能力升级

#### 必备输入

1. discovery 结论或已有需求文档

#### 阶段输出

1. 需求文档
2. 功能拆解
3. 范围边界
4. 优先级列表
5. 可交付目标

#### 可跳过规则

以下情况可弱化或跳过：

1. 老项目只做小范围 bugfix
2. 当前迭代目标已由现有 ticket 明确界定

但即使跳过，也应形成简化任务定义。

#### 主导 agent

- `Planner Agent`

#### 检查角色

- `Reviewer Agent`

#### 退出条件

1. 功能范围明确
2. 优先级和边界可执行
3. 后续可进入技术方案设计或直接进入实现准备

---

### 5.4 `architecture`

#### 阶段目标

形成技术方案、模块边界、关键接口、数据流和重要工程决策。

#### 常见进入场景

1. 新项目准备落技术方案
2. 老项目准备做较大架构变更
3. 全栈需求需要前后端协同设计

#### 必备输入

1. 需求说明
2. 项目技术栈和约束
3. 已有系统结构信息

#### 阶段输出

1. 技术方案说明
2. 模块划分
3. 接口契约草案
4. 风险点与替代方案
5. 技术决策记录

#### 可跳过规则

以下情况可弱化：

1. 小型改动且影响范围有限
2. 老项目已有成熟架构，只需局部延续

#### 主导 agent

- `Architect Agent`

#### 检查角色

- `Reviewer Agent`
- 必要时附加 `Security Reviewer Agent`

#### 退出条件

1. 技术路径可执行
2. 模块边界和关键契约清晰
3. 主要实现风险已识别

---

### 5.5 `scaffold`

#### 阶段目标

完成工程骨架、项目规范、基础目录和 AI 控制层的准备。

#### 常见进入场景

1. 新项目需要初始化工程骨架
2. 老项目需要完成 `.ai-first/` 注入
3. 团队准备建立项目规范和基础脚手架

#### 必备输入

1. 需求与技术方案
2. 项目组织方式
3. 团队标准和复用配置

#### 阶段输出

1. 工程基础结构
2. `.ai-first/` 控制层
3. 初始标准与模板
4. 基础 skill 配置
5. 初始项目状态文件

#### 主导 agent

- `Builder Agent`

#### 检查角色

- `Reviewer Agent`

#### 退出条件

1. 项目已具备可持续迭代基础
2. AI 控制层已可运行
3. 可进入持续构建阶段

---

### 5.6 `build`

#### 阶段目标

进行持续开发、功能实现、缺陷修复和全栈闭环交付。

#### 常见进入场景

1. 新项目进入主开发
2. 老项目持续迭代
3. 全栈研发负责一个功能闭环

#### 必备输入

1. 当前任务定义
2. change scope
3. 相关标准和上下文
4. 必要的需求或方案信息

#### 阶段输出

1. 代码改动
2. 文档改动
3. 任务完成说明
4. 知识同步候选项

#### 阶段机制

1. 任务应按业务功能闭环组织
2. 需要声明影响范围
3. 允许多人并行，但要检查模块冲突
4. 老项目全栈模式下要同时关注前后端契约一致性

#### 主导 agent

- `Builder Agent`

#### 检查角色

- `Reviewer Agent`
- 根据任务类型追加专项 reviewer

#### 退出条件

1. 当前任务功能已完成
2. 对应检查已进入或准备进入 `qa`

---

### 5.7 `qa`

#### 阶段目标

验证本轮交付是否达到质量要求。

#### 常见进入场景

1. build 阶段任务完成
2. 准备发布前验证
3. 老项目修复后需要回归

#### 必备输入

1. 代码改动
2. 测试结果或检查结果
3. 风险点清单

#### 阶段输出

1. 质量检查结果
2. bug 和风险列表
3. 安全扫描结果
4. 回归建议
5. 是否允许进入 release 的结论

#### 检查项

1. 代码逻辑
2. 功能一致性
3. 安全风险
4. 文档更新
5. 知识同步状态

#### 主导 agent

- `Reviewer Agent`

#### 专项角色

- `Security Reviewer Agent`
- `QA Agent`

#### 退出条件

1. 关键问题已处理
2. 风险已可接受或已记录
3. 满足 release 进入条件

---

### 5.8 `release`

#### 阶段目标

完成发布准备、最终检查和交付确认。

#### 常见进入场景

1. 本轮迭代准备上线
2. 老项目修复或功能交付准备发布

#### 必备输入

1. QA 结论
2. 发布说明
3. 关键变更摘要

#### 阶段输出

1. 发布检查清单
2. 发布说明
3. 已知风险记录
4. 上线结论

#### 主导 agent

- `Release Agent`

#### 检查角色

- `Reviewer Agent`

#### 退出条件

1. 发布 gate 通过
2. 关键同步项已确认
3. 项目进入 `operate`

---

### 5.9 `operate`

#### 阶段目标

支持上线后维护、监控反馈、问题处理和知识更新。

#### 常见进入场景

1. 项目已上线
2. 团队持续维护线上系统
3. 老项目长期处于运维迭代状态

#### 必备输入

1. 发布信息
2. 运行反馈
3. 线上问题或新需求

#### 阶段输出

1. 运营反馈记录
2. 问题归因
3. 优化建议
4. 下一轮迭代输入

#### 主导 agent

- `Team Lead Agent`

#### 检查角色

- `Reviewer Agent`

#### 退出条件

1. 有明确的后续优化方向或新增需求
2. 可进入 `evolve` 或再次回到 `build`

---

### 5.10 `evolve`

#### 阶段目标

基于已有项目运行结果和团队目标，规划下一轮迭代。

#### 常见进入场景

1. 新需求进入
2. 老项目准备进入下一轮版本
3. 平台希望推动经验沉淀升级为标准或 skill

#### 必备输入

1. operate 阶段反馈
2. 当前项目状态
3. 团队沉淀与问题记录

#### 阶段输出

1. 下一轮目标
2. 优先级调整
3. 经验升级候选项
4. 新任务集

#### 主导 agent

- `Team Lead Agent`

#### 检查角色

- `Reviewer Agent`

#### 退出条件

1. 已形成下一轮可执行任务
2. 平台能重新进入 `discovery`、`spec` 或 `build`

---

## 6. 新项目与老项目的阶段进入策略

### 6.1 新项目默认进入策略

新项目通常从以下入口开始：

- `idea`
- `discovery`
- `spec`

具体进入点由输入完整度决定：

1. 只有想法时进入 `idea`
2. 有明确目标但需求未成型时进入 `discovery`
3. 已有较完整需求文档时进入 `spec`

### 6.2 老项目默认进入策略

老项目通常不从 `idea` 开始，而从以下阶段切入：

- `build`
- `qa`
- `operate`
- `evolve`

平台在 `adopt` 后需要先做一次阶段识别，再决定进入点。

### 6.3 老项目全栈场景的特殊规则

1. 前端和后端属于同一项目状态机，不拆分为两套生命周期
2. 任务进入 `build` 时应按功能闭环组织
3. `qa` 时需检查前后端接口契约和联调一致性
4. `evolve` 时要能同时沉淀前端、后端和 `fullstack` 标准

## 7. 阶段判断规则

平台应基于项目快照进行阶段判断，建议使用“阶段 + 置信度”模型。

### 7.1 判断输入

1. 仓库结构
2. 文档存在情况
3. 需求文档和设计文档情况
4. 测试与发布配置情况
5. 最近代码活跃区域
6. issue / task / 变更记录
7. 用户显式输入目标

### 7.2 判断输出

建议输出结构如下：

```ts
type StageAssessment = {
  currentStage: ProjectStage
  confidence: number
  reasons: string[]
  alternativeStages: ProjectStage[]
  blockers: string[]
  missingArtifacts: string[]
}
```

### 7.3 判断原则

1. 若多个阶段都可能，应返回主阶段和备选阶段
2. 老项目若处于长期维护态，可优先判为 `operate` 或 `build`
3. 小范围改动不必强制走完整上游阶段，但必须有最小任务定义和检查

## 8. 可跳过、可复用、可生成机制

这是状态机的重要柔性约束。

### 8.1 `Generate`

适用场景：

1. 新项目缺少现成资料
2. 团队希望快速起草需求、方案、文档
3. 平台需提供起步模板

### 8.2 `Reuse`

适用场景：

1. 用户已有需求文档、架构文档或 wiki
2. 老项目已有成熟资料
3. 团队希望沿用现有标准

要求：

1. 平台需给出建议和差异分析
2. 用户可采纳或拒绝建议

### 8.3 `Skip`

适用场景：

1. 小步快跑的老项目修复
2. 目标和方案已足够明确
3. 团队主动接受后续风险

要求：

1. 平台必须记录跳过原因
2. 平台必须标记潜在风险
3. reviewer 在后续阶段应补充检查

---

## 9. 阶段与 agent 路由关系

建议第一版建立以下主导映射：

| 阶段 | 主导 agent | 主要职责 |
|---|---|---|
| `idea` | `Intake Agent` | 收敛目标与初始边界 |
| `discovery` | `Planner Agent` | 澄清需求和场景 |
| `spec` | `Planner Agent` | 形成需求定义与任务拆解 |
| `architecture` | `Architect Agent` | 技术方案与关键决策 |
| `scaffold` | `Builder Agent` | 初始化项目骨架与控制层 |
| `build` | `Builder Agent` | 实现功能和推进任务 |
| `qa` | `Reviewer Agent` | 质量、安全、逻辑与同步检查 |
| `release` | `Release Agent` | 发布检查与交付确认 |
| `operate` | `Team Lead Agent` | 维护、反馈和问题跟踪 |
| `evolve` | `Team Lead Agent` | 规划下一轮迭代 |

说明：

1. 第一版坚持“单阶段单主导 agent”为主
2. 其他 agent 作为辅助角色参与
3. 所有阶段都应保留 reviewer 视角

---

## 10. 阶段与检查机制关系

状态机不能只管推进，还必须管收口。

### 10.1 阶段结束检查

每个阶段在退出前至少检查：

1. 必备产物是否存在或已被明确跳过
2. 风险是否被记录
3. 是否具备进入下一阶段的最小条件
4. 是否需要同步知识或 wiki

### 10.2 重点阶段的额外检查

- `architecture`
  - 关键接口是否清晰
  - 风险是否显式

- `build`
  - 是否声明 change scope
  - 是否影响前后端契约

- `qa`
  - 是否覆盖逻辑、安全、知识同步检查

- `release`
  - 是否完成发布前 gate

---

## 11. 与知识同步的关系

状态机必须和知识同步联动。

### 11.1 推荐触发点

以下阶段退出时优先触发同步建议：

1. `spec`
2. `architecture`
3. `build`
4. `qa`
5. `release`
6. `evolve`

### 11.2 推荐同步内容

1. 项目知识
2. wiki
3. 技术决策记录
4. 标准候选项
5. skill 候选项

---

## 12. MVP 状态机范围

第一版不必把所有阶段做成完全自动化，但状态机必须先定义完整。

### 12.1 MVP 必须重点支持的阶段

1. `spec`
2. `architecture`
3. `scaffold`
4. `build`
5. `qa`
6. `operate`

### 12.2 MVP 可弱化支持的阶段

1. `idea`
2. `discovery`
3. `release`
4. `evolve`

说明：

1. 新项目第一版可以从 `spec` 开始强化体验
2. 老项目第一版重点支持 `build`、`qa`、`operate`
3. 虽然有阶段轻重之分，但统一状态机仍应完整保留
