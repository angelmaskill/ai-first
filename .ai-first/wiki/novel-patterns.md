# Novel Framework Patterns
**Last updated**: 2026-04-27
**Source**: .ai-first/knowledge/KNOW-003-novel-patterns.md

Three novel framework-level patterns emerged during the AI-First scaffold design.

## Pattern 1: Rules Locking

**File**: `.ai-first/locks/rules.lock`

During execution stages (build, qa, release), project standards and stable knowledge are locked as read-only. Code must conform to rules — rules do not change to fit code.

**Lifecycle**:
- **Locked**: build → qa → release (enforced)
- **Unlocked**: evolve → discovery → spec → architecture → scaffold (editable)

**Why novel**: Traditional projects allow standards to drift alongside code. The lock enforces a strict separation between "define the rules" and "follow the rules" phases, preventing the common pattern of weakening standards to accommodate sloppy code.

## Pattern 2: Append-Only Timeline

**File**: `.ai-first/logs/timeline.md`

All state mutations (stage transitions, scan results, review verdicts, task completions) are logged to an append-only timeline. Each entry has a timestamp, category tag, and one-line summary.

**Why novel**: This is not a traditional log (which is ephemeral and noisy) or a database (which allows updates and deletes). The append-only timeline provides an immutable audit trail that requires no query engine — `grep` and `tail` are the only tools needed. Contrast with CI logs (lost after retention period) or issue trackers (allow edits, deletions, status changes).

## Pattern 3: Dedicated State-Updater Agent

**File**: `.claude/agents/state-updater-agent.md`

A single haiku-tier agent has exclusive write access to all project state: symlink updates, project.yml mutations, task status changes, and timeline appends. All other agents must request state changes through this agent.

**Why novel**: This eliminates an entire class of bugs: concurrent state mutations, inconsistent symlink ↔ project.yml state, partial updates. Rather than every agent calling `ln -sf` or editing YAML directly, there is a single narrow-waist API for state. The exclusive flag in routing.yml (`exclusive: true`) ensures no other agent can perform this intent.
