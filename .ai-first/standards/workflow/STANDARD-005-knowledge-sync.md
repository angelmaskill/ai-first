# Knowledge Synchronization

After every build task, changed files must be checked against the
knowledge base for staleness. This prevents documentation rot.

## Rule

Knowledge-sync-agent must run after every build task completes.
It must not be skipped, even for single-line changes.

## Sync Trigger Categories

| File Pattern | Sync Category | What to Check |
|-------------|--------------|---------------|
| routes/, controllers/, api/ | api-contract | API docs, endpoint signatures |
| models/, schemas/, types/ | data-model | Data model docs, schema diagrams |
| config files (.yml, .yaml, .json) | config | Deployment docs, env var docs |
| src/*/index.ts, architecture | architecture | Architecture docs, ADRs |
| auth, middleware/auth | security | Security standards, auth flows |
| package.json, Dockerfile | dependency | Dependency docs, setup guide |

## Sync Event Lifecycle

```
suggested → pending → confirmed → resolved
                         ↘ dismissed
```

- **suggested**: Auto-generated by knowledge-sync-agent
- **pending**: Acknowledged by human, waiting for action
- **confirmed**: Update applied, knowledge item revised
- **dismissed**: False positive or intentionally skipped

## Minimum Knowledge Base

Every project must have at minimum:
- INDEX.md with all knowledge item references
- Project overview knowledge item
- Knowledge items for any custom framework patterns

## Stage Exit Check

Before advancing stages, all sync events must be either `confirmed`
or `dismissed`. No `suggested` or `pending` events may remain.
