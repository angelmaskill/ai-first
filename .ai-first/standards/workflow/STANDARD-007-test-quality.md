# Test Quality and Coverage Standards

## Rule

Every code change must be accompanied by appropriate tests. No untested
code may enter production. The test pyramid (unit > integration > e2e >
smoke) must be respected, and critical paths must have smoke test coverage.

## Test Pyramid

| Layer | Coverage Target | Max Runtime | When to Run |
|-------|----------------|-------------|-------------|
| **Unit** | ≥ 70% line coverage | < 30s | Every commit |
| **Integration** | Core APIs ≥ 90% | < 2min | Every push |
| **E2E** | Critical user flows | < 5min | Pre-merge |
| **Smoke** | All P0 + P1 paths | < 5min | Post-deploy |

## Test:Source Ratio

The project must maintain a minimum test:source file ratio:

| Phase | Minimum Ratio | Target Ratio |
|-------|-------------|-------------|
| MVP | 0.3 | 0.5 |
| Production | 0.5 | 0.8 |
| Regulated | 0.8 | 1.0 |

Current target for AI-first: **0.5** (28 tests / ~60 source files = 0.47).

## Test Quality Requirements

### Unit Tests

- Pure functions: test all branches (happy path + error path + edge cases)
- Functions with side effects: mock all external dependencies
- React components: test render output + user interactions + error states
- Never: hit real APIs, databases, or filesystems in unit tests

### Integration Tests

- Test real module interactions (no mocks between modules)
- Mock only external boundaries (HTTP, DB, filesystem at the adapter layer)
- Each API endpoint must have: success (200/201), validation error (400),
  auth error (401/403), not found (404), server error (500)

### E2E Tests

- Cover the top 3 user journeys (happy path only)
- Run against a staging environment (never production)
- Must be idempotent (can run multiple times without side effects)

### Smoke Tests

- All P0 paths: must pass or deployment is blocked
- All P1 paths: must pass or hotfix is scheduled
- Generated by smoke-case-agent after every release preparation
- Stored in `.ai-first/reports/smoke-cases-*.md`

## Test Generation

When new code is added without tests:

1. **test-generator skill** analyzes the source and produces test skeletons
2. Test skeletons are written to the appropriate test file
3. Reviewer-agent testing gate verifies that tests exist and are meaningful
4. CI pipeline enforces coverage thresholds

## Test→Production Boundary

The following must NEVER appear in non-test production code:

| Category | Examples | Detection |
|----------|---------|-----------|
| Localhost URLs | `http://localhost:3000` | bug-scan 1.6 |
| Test domains | `*.dev.*`, `*.test.*`, `staging.*` | bug-scan 1.6 |
| Test credentials | `test@`, `demo@`, `password: test` | bug-scan 1.6 |
| Debug flags | `DEBUG=true`, `debugMode=true` | bug-scan 1.6 |
| Hardcoded dev ports | `:3000`, `:4200`, `:5173` | bug-scan 1.6 |

These are checked by the enhanced bug-scanner skill (Phase 1.6).

## Smoke Test Case Generation

The smoke-case-agent runs during release preparation:

1. Identifies entry points, API surface, service dependencies, data layer
2. Classifies paths into P0/P1/P2 priority
3. Generates structured smoke test cases with expected responses
4. Produces `scripts/smoke-test.sh` for CI/CD integration

At least 60% of P0 tests and 30% of P1 tests should be automated in CI.

## Verification

- `npx vitest run` must pass for all commits
- `npx vitest run --coverage` must show ≥ target ratio
- `bash scripts/smoke-test.sh` must pass before production deployment
- Pre-commit hook runs secret detection (catches test→prod leaks early)
- CI pipeline enforces coverage thresholds on push
