# Codex Prompt v0 (Rich Context In, Lenient Out)

> 这是 §4.2 prompt v0 的参考模板，由 `renderPromptV0(bundle)` 在代码里渲染。
> 本文件作为**人类可读的契约文档**——让团队成员和 Codex 都清楚任务上下文包长什么样、
> 以及"宽容产出"原则（不要求 Codex 填 schema）。

## 设计原则（§0）

- **丰饶上下文 in**：把任务、scope、domain 上下文、相关规范、验收条件、当前阶段都喂给 Codex。
- **宽容产出 out**：末尾请 Codex 用一段话简述改动，**不要求 JSON / 固定字段**。
  产出报告由工具侧基于 git change set、acceptance runner、Codex 进程状态客观收集。

## 模板（由 renderPromptV0 生成）

```text
# 任务
{task.title}

{task.description}

# 改动范围（请只在这些路径内改动）
- frontend: {scope.frontendPaths}
- backend:  {scope.backendPaths}
- algorithm: {scope.algorithmPaths}
- data:     {scope.dataPaths}
- docs:     {scope.docsPaths}

# 相关 domain 上下文
{对每个 domainContext: kind + paths + 技术栈 + 测试/构建命令}

# 相关规范（团队约定，请遵守）
{对每个 standardDigest: [id] title (severity) — must 条目}

# 验收条件（完成后这些应当成立）
{对每条 acceptanceCriteria: - [id] description (check kind)}

# 当前阶段
{stage}

---
请完成上述任务。你可以自由决定实现方式。
完成后，用一段话简述：改了哪些主要文件、跑了什么验证、有没有遗留问题或风险。
不需要输出 JSON 或任何固定格式。
```

## 工具侧客观收集（不依赖运行时自报）

| ExecutionReport 字段 | 来源 |
|---|---|
| `status` (done/review_pending/blocked) | 三态机：进程状态 + scope 违规分级 + 必需验收 |
| `outcomeReason` | 诊断原因（不参与决策） |
| `filesChanged` | `GitChangeSet.trackedChanges ∪ untrackedChanges` |
| `scopeViolations` | `classifyScopeViolations(filesChanged, scope, domains)` → risk/review/block |
| `acceptanceResults` | `runAcceptancePlan(criteria, allowed)`（只跑登记命令） |
| `runtimeStdout/Stderr/ExitCode` | `PromptExecutor.executePrompt()` 原样留存 |
| `naturalLanguageSummary` | `extractTailSummary(stdout)`，best-effort，可能为空 |
| `risks / blockers` | 越界项 + 验收失败项 + 进程失败原因 |
| `knowledgeSyncNeeded` | `filesChanged` 非空则 true |

## 越界分级（§4.3.3 / ADR-002）

| 越界目标 | severity | 影响 status |
|---|---|---|
| scope 外测试/文档/helper | risk | 不影响（done 仍成立，仅进 risks） |
| 同 domain 但 scope 外的源码 | review | 升级 review_pending |
| 跨 domain 契约/API/schema | review | 升级 review_pending |
| security / release / CI / `.ai-first/standards/` / `project.yml` | block | 升级 blocked |

> Codex 自由的是**任务范围内的代码实现**；触及安全/发布/规范基础设施必须拦——
> 这与"不限制 Codex 发挥"不冲突。
