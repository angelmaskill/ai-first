## 7. CSP Readiness (Inline Styles)

### Result: ADVISORY (Medium)

**Finding**: 81 inline `style={{...}}` objects across 9 component files. This is consistent and intentional -- the project uses React inline styles as its primary styling approach (supplemented by CSS classes `card`, `badge`, `stat-value`, `stat-label` from `global.css`).

**CSP implications**:
- React's `style` prop generates inline `style="..."` attributes in the DOM.
- A strict Content-Security-Policy would require `style-src 'unsafe-inline'`, which weakens CSP protection.
- No `nonce` or `hash`-based style allowlisting is currently configured.

**Files with inline styles** (count of style props):
| File | Style props (approx) |
|------|---------------------|
| Dashboard.tsx | ~25 |
| StageIndicator.tsx | ~15 |
| HealthCard.tsx | ~12 |
| RiskList.tsx | ~8 |
| SyncStatus.tsx | ~10 |
| ActionCard.tsx | ~8 |
| Timeline.tsx | ~10 |
| LanguageSwitcher.tsx | ~3 |
| App.tsx | ~3 |

**Mitigating factors**:
- React's `style` prop does not allow `!important`, `@import`, or URL values that could be exploited for script injection.
- No dynamic CSS-in-JS library (styled-components, emotion) is used -- the values are all static design tokens or component-local expressions.
- The 81 inline styles are all authored by developers, not generated from user input.

**Recommendations**:
1. (Medium) Add a CSP meta tag with `style-src 'self' 'unsafe-inline'` as a baseline, and document the constraint.
2. (Future) If CSP hardening becomes a priority, migrate inline styles to CSS Modules or a zero-runtime CSS-in-JS library (e.g., Vanilla Extract, Panda CSS) to enable `style-src 'self'` without `unsafe-inline`.

## 11. Additional Observations

### LanguageSwitcher Inline Event Handlers

**File**: `src/frontend/components/LanguageSwitcher.tsx:24-31`

```tsx
onMouseEnter={(e) => {
  e.currentTarget.style.background = "var(--color-cream)";
  e.currentTarget.style.borderColor = "var(--color-indigo)";
}}
onMouseLeave={(e) => {
  e.currentTarget.style.background = "var(--color-warm-white)";
  e.currentTarget.style.borderColor = "var(--color-sand)";
}}
```

**Assessment**: Acceptable. These modify `style.background` and `style.borderColor` on the button element itself using CSS variable references -- no user input, no URL values, no script context. Not exploitable. However, for CSP `script-src 'unsafe-inline'` avoidance, these could be replaced with CSS `:hover` pseudo-class, eliminating the inline event handlers.

### No Content-Security-Policy Configuration

No CSP meta tag or HTTP header is configured. Given the static nature of the dashboard and zero external resource loading, adding a CSP would be a defense-in-depth measure. Suggested baseline:
```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; style-src 'self' 'unsafe-inline'; script-src 'self'; img-src 'self' data:; font-src 'self';">
```

Note: `style-src 'unsafe-inline'` is currently required due to the 81 inline style props (see Section 7).
