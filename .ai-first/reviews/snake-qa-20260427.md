## 1. Logic Gate -- PASS (3 findings)

### 1a. Double-draw on gameover (Suggestion)

**File**: `/tmp/snake-game-project/src/engine.ts`, lines 42-49

The `gameLoop` function calls `draw(state)` unconditionally at line 42, then calls `draw(state)` again at line 46 when `phase === 'gameover'`. This results in a redundant frame render on the game-over transition. It is a single extra frame and causes no visual glitch (rendering is idempotent), so this is purely cosmetic.

```
// Current:
draw(state);
if (state.phase !== 'gameover') {
  requestAnimationFrame(gameLoop);
} else {
  draw(state);          // <-- redundant with line 42
  showGameOver(state.score);
}
```

### 1b. Wall-collision head rendering (Suggestion)

**File**: `/tmp/snake-game-project/src/engine.ts`, lines 61-64

When a wall collision occurs, `state.snake = moved` is set *before* returning, placing the head at an out-of-bounds coordinate (e.g., `{x: -1, y: 10}`). The final `draw()` call then attempts to render this head segment via `fillRect` with negative coordinates. The browser silently clips the rectangle, so the snake head disappears on the death frame. This is a minor visual artifact that most players will not notice. If intentional (the head "leaves" the grid), it could be documented; otherwise the head could be clamped to the last valid position before drawing.

### 1c. Game loop robustness (Positive Finding)

The elapsed-time gating (`elapsed >= speed`) combined with `performance.now()` reset on unpause (line 88 of engine.ts) correctly prevents time-skip exploits. The direction 180-degree reversal guard (`isOpposite`) combined with single-variable `nextDirection` correctly prevents rapid-key circumvention. The state machine transitions are well-constrained: `playing <-> paused` and `playing -> gameover -> (restart) -> playing`. No invalid transitions are reachable.

### 3b. `directionFromKey` placement (Suggestion)

**File**: `/tmp/snake-game-project/src/snake.ts`, lines 32-38

`directionFromKey` maps keyboard event keys to `Direction` values. It is consumed only by `input.ts`. Its placement in `snake.ts` creates a slight cohesion violation: key mapping is a UI/input concern, not a snake-logic concern. However, it is a pure function with no side effects, so the impact is negligible. Options if refactoring: move to `input.ts` (colocation with consumer), `types.ts` (as a lookup table), or leave as-is (acceptable for this project size).
